# 肠菌健康管理系统 · 对接方案

> 前置：需求见 [`requirements.md`](requirements.md)。契约基线：**`contract-v1.0`**（`../../interfaces/`）。
> 读者：我方（排期/缺口）+ SI 集成方（对接什么、怎么分工）。2026-06-11。

## 0. 方案一句话

**每家门店一台 Pinea 盒子，AI 能力（评估推荐/报告解读/问答/转写提取）在盒内建一次、多端 UI 共用，健康数据不出门店；业务应用（CRM/会员/支付/内容）、客户档案、外呼/推送通道在 SI 侧；总部对门店的「权重影响」走 skill 资产包版本下发。**

## 1. 部署拓扑与责任边界

```
   总部云（盒外 · SI/客户）                 每家门店（盒内 = Pinea 盒子 :18800）
┌───────────────────────────┐          ┌─────────────────────────────────┐
│ 端三 总部后台（产品库/权重）   │ skill 包  │ ① 能力面  chat·embeddings·STT     │
│ 跨端 运营系统（推送/外呼/客服） │ ───────→ │ ② Agent 面  会话/异步（T2.5）      │
│ 客户档案 CRM / 知识库         │  版本下发  │ ③ Skills  产品库·权重·规范库·阈值表 │
└───────────────────────────┘          └─────────────────────────────────┘
        ↑ 结构化结果回传（SI 应用层负责）            ↑ HTTP（OpenAI SDK 直连）
                                        端一 设备终端 UI · 端二 店员工作台（SI 应用）
```

切分原则（对客户讲清楚的一句话）：**模型推理在盒内，业务系统在盒外**。

| | 盒内（Pinea 交付） | 盒外（SI/客户负责） |
| --- | --- | --- |
| 算什么 | 评估推荐、报告解读、摘要、问答生成、转写、信息提取 | — |
| 存什么 | skill 资产（产品库/权重/规范库/阈值表，只读下发） | 客户档案、问卷库、知识库、操作记录、推送/外呼记录 |
| 通道 | 仅 `:18800` HTTP（LAN 外放需 Bearer + TLS） | 微信/短信推送、电话外呼线路、Web/App 前端 |
| 明确不做 | 电话级实时语音闭环（§4.4）、客户档案存储、**支付/会员交易**（涉资质，纯 SI 侧） | — |

两点补充：

- **业务 app 可上盒（机会项）**：设备终端的业务应用可作为本地 web 服务跑在盒子上（触屏即门店终端），一台机器 = 业务终端 + AI 推理，离线叙事更完整。责任切分不变：app 进程归 SI，盒子保证宿主环境 + `:18800` 能力面；进程/端口/资源配额在交付时约定。**app 开发归属是头号商务问题**（requirements §5#1），拍板前不排业务线工期。
- **离线即卖点**：断网可推理是盒子天然属性，健康数据不出门店是合规叙事核心。

## 2. AI 能力线 → 接口映射（能力建一次，多端共用）

端一/端二的 AI 功能重叠（报告解读、问答两端都要）——按「端」建设会重复造轮子，**按能力线组织**：

档位标注：✅=已实现；T1/T2/T3/T1.5=契约已定按排期；spike=先实测定承诺。

### 2.1 评估推荐线（旗舰 · 一期）

| 环节 | 落点 | 对接方式 |
| --- | --- | --- |
| 问卷/体检数据输入 | SI→盒 | SI 组装结构化 JSON（schema 见 `phase1-design.md` §3） |
| 评估+推荐+科普输出 | 盒内 | ① chat + `json_schema` ✅ 单次调用出全量结构化结果；schema 合规由网关保障（T1.5） |
| 离线推理 | 盒内 | 本地模型天然离线；装机验收含拔网线实测 |
| 店员调整/存档/绑定档案 | 盒外 | SI 应用层；存档带 `asset_version` 可追溯 |

消费端：端一设备（主）。详设：[`phase1-design.md`](phase1-design.md)。

### 2.2 报告解读线（二期）

| 环节 | 落点 | 对接方式 |
| --- | --- | --- |
| 报告上传 + OCR 字段识别 | **spike 定** | 盒内方案=① 视觉理解（minicpm-v 字段抽取，样张 ≥20 实测定承诺档位）；不达标 SI 侧 OCR 兜底 |
| 指标结构化提取 | 盒内 | ① chat + `json_schema`：{多样性指数, 优势菌属, 失衡菌属, …} |
| 异常判定 | 盒内+规则 | **阈值表（规则）判异常，模型只写说明/科普**（§4.1） |
| 摘要/高亮文案 | 盒内 | ① chat + 规范库措辞约束 |
| **纵向对比**（多次报告趋势） | 盒内+SI | SI 取历史结构化报告随请求送入，① 生成对比解读；比对口径待 requirements §5#8 |
| 批量 20 份 | 盒内 | `X-Pinea-Priority: batch`（T2）排队，不挤交互（§4.5） |

消费端：端一设备 + 端二工作台（同一能力，两套 UI）。

### 2.3 问答线（二期）

| 环节 | 落点 | 对接方式 |
| --- | --- | --- |
| 知识库向量化/检索 | SI 侧 | ① `/v1/embeddings` ✅ + SI 向量库（知识库是客户资产，检索可控可溯源） |
| 答案生成 | 盒内 | ① chat + 规范库约束 |
| 消费者版护栏（百问百答） | 盒内+SI | 拒答/转店员范围客户圈定（§5#9）+ 禁用词扫描；店员版护栏低一档 |
| 反馈回流 | 盒外 | SI 标记，知识库维护 |

消费端：端一「百问百答」（消费者）+ 端二「售后问答」（店员）——**同一能力，两档护栏**。

### 2.4 转写提取线（三期）

| 环节 | 落点 | 对接方式 |
| --- | --- | --- |
| 到店沟通转写 | 盒内 | ① `/v1/audio/transcriptions`（T1）；≤25MB/30min，长录音切分，批量走夜间 batch 档 |
| 外呼/来电转写 | 盒外为主 | 录音在云端产生，就近云端转写；或回传盒子批量处理（三期定） |
| 关键信息提取 | 盒内 | 转写文本 → ① chat + `json_schema`：{核心需求, 痛点, 诉求, 意向产品, 到店意愿} |
| 档案归档 | 盒外 | SI 档案系统 |

### 2.5 盒外清单（AI 不参与或仅可选增值）

CRM 建档/分类/标签、会员/支付/核销、产品与菌株介绍页、科普文章运营、画像筛选与定向推送、7×24 云客服与自动外呼（**电话级实时语音闭环明确不做**，盒子角色=通话后转写+提取）。可选增值：文章/话术初稿用盒子 ① 生成、客户自动打标签。

## 3. 核心链路 A→B 演进与总部下发

- **A 方案（一期 · 只靠 ① ✅）**：SI 组装资产进 prompt → ① chat + `json_schema` 出结构化结果。资产由 SI 应用层管理，但**从第一天就版本化**。
- **B 方案（二期 · ③+②，对齐 T3）**：资产打包成行业 skill **`gut-recommend`**（产品库/权重/规范库/阈值表/输入 schema 随包只读下发，SKILL.md 固化工序），② `/v1/agent/chat/completions` + `"skill":"gut-recommend"` 确定性调用。SI 切换只改请求体，schema 不变。验收：金标准样例集 ≥20 实测 N/N。
- **总部下发 = skill 包版本发布（三期，解端三与离线的矛盾）**：总部改权重 → 打出 `gut-recommend vX.Y` → 门店 updater 拉取 → ③ 重装即生效；断网用旧版，恢复追平。**版本号进每条推荐存档**，可追溯对账。

## 4. 五个工程判断（提前定，避免返工）

1. **异常判断不交给模型**：指标越界由阈值规则表判（确定性、可审计），模型只把已判定的异常写成人话+科普。健康场景里"判断错"和"说得不好"是两种责任。
2. **RAG 在 SI 侧**：盒子给 embeddings + chat，向量库/检索在 SI——知识库本就是客户资产，答案溯源由检索层保证。
3. **盒子不存客户档案**：推理输入即用即走（① 无状态），输出由 SI 入库。② 会话注意 workspace 跨 session 共享警告（`agent-api.md` §4），跨最终用户隔离由 SI 负责。
4. **不做电话级实时语音闭环**：电信级延迟+呼叫通道资质均在云侧。第一次见客户就讲死，防期望错位。
5. **批量按批量类对待**：批量解读/夜间转写声明 `X-Pinea-Priority: batch`，撞闸门按 `429`+`Retry-After` 排队重试（quickstart 给样例代码），店员当面交互永远优先。

## 5. 分期计划（客户分期为主轴，对齐 build-plan T 序列）

分期原则：**按客户价值切，每期独立可验收、可收款**；平台 T 序列并行推进，后期消费前期落地的端点。

| 期 | 范围 | 平台依赖 | 状态 |
| --- | --- | --- | --- |
| **一期 · 评估推荐线 + 设备端落地** | A 方案全链路 + 调整/存档对接 + T-D 交付件首次实战 | 仅 ① chat+structured outputs（✅）——**零新端点依赖，关键路径在客户数据资产** | **详设：[`phase1-design.md`](phase1-design.md)** |
| **二期 · 报告解读线 + 问答线 + skill 化** | 解读/摘要/高亮+纵向对比、双护栏问答、OCR 落地；`gut-recommend` skill 化，A 切 B | T1 STT、T1.5、T2/T2.5、T3；OCR spike 结论 | 待一期 M2 后详设 |
| **三期 · 总部下发 + 转写线 + 多门店** | skill 包版本发布 + updater、转写提取线、多门店复制 | ③ 版本化（T3）、updater（T-D #6） | 待二期 |

每期通用节奏：schema 定稿 → mock 给 SI 并行 → 客户资产入库 → 真模型实测出数 → 调优达线 → 交付验收。一期跑成型，二三期复制。

## 7. 承诺与合规口径（见客户前对齐）

- **SLA 边界**（`model-gateway.md` §6.6）：承诺契约行为 + 进程可用性 + 样例集实测 N/N；**不承诺模型输出质量 100%**。唯一例外：schema 合规率随 T1.5 落地可承诺 100%（契约行为）。
- **措辞红线**：「健康评估参考 / 调理建议」，规避「诊断 / 治疗 / 诊疗方案」（requirements §5#2）——规范库固化约束 + 禁用词扫描双保险；医疗广告/器械合规审查责任在客户方。
- **数据合规卖点**：推理本地化、健康数据不出门店、① 无状态不留存、删除语义明确。外呼录音告知义务、个保法口径在盒外，由客户/SI 承担。
