# 肠菌方案 · 典型使用场景与实现流程

> 配套：能力线映射见 [`integration-plan.md`](integration-plan.md) §2，一期详设见 [`phase1-design.md`](phase1-design.md)。
> 用途：M0 评审 / 给 SI 与客户讲方案。每个场景三段：**看到什么（人）→ 流程（步骤）→ 怎么实现（端点与分工）**。标注一/二/三期。

## 场景 1 · 到店评估推荐（一期 · 旗舰）

**看到什么**：顾客王女士进店，店员引导她在智能终端上花 3 分钟做肠道健康问卷（症状、病史、用药、调理需求）。提交后屏幕先出现评估结论，随后逐条出现 2~3 个推荐产品，每条带"为什么推荐"（症状匹配 + 菌属匹配）和注意事项，下方附异常项科普。店员看一眼，把其中一款换成店内现货，确认后整单存入王女士档案。

**流程**：

1. 终端 UI（SI app）收集问卷 → 组装结构化 JSON（`profile`/`medical_history`/`gut_symptoms`/`goals`…）；
2. SI app 本地跑**规则表**：红旗清单判定（未命中）→ 候选预筛（按症状筛出 ≤15 个 SKU，附权重要点）；
3. 调盒子 `POST /v1/chat/completions`，`response_format: json_schema` + `stream: true`；
4. SSE 流式返回：先 `conclusion` 块（UI 立即展示），再 `recommendations[]` 逐条渲染，`education[]` 殿后；
5. 店员替换一款产品（操作留痕）→ SI app 把「问卷快照 + AI 推荐 + 调整记录 + `asset_version`」写入客户档案（SI 侧数据库）。

**实现**：盒内只有第 3~4 步（① 能力面，T0 已绿）；规则在 SI 侧执行但**规则内容是版本化资产**；模型 `keep_alive` 常驻保证 P95 ≤ 30s。完整无网络依赖。

## 场景 1b · 红旗命中（一期 · 安全底线）

**看到什么**：李先生问卷勾选了"近期便血"。屏幕不出现任何产品推荐，而是显示："您的部分情况建议优先就医检查……"（客户认可的标准话术），并提示店员协助。

**流程**：SI app 规则表第一步即命中红旗 → **直接终止，不调模型**（或调模型仅生成就医建议的礼貌表述，`red_flag.triggered=true`，schema 强制 `recommendations` 为空）→ 存档记录红旗触发。

**实现**：确定性规则，不依赖模型判断——这是能对"安全"做硬承诺的唯一方式。验收要求红旗样例 3/3 触发且零产品推荐。

## 场景 2 · 断网照常营业（一期 · 离线卖点）

**看到什么**：商场网络故障半天，隔壁柜台的云端收银一直转圈，这家店的评估推荐照常做，体验无任何变化。

**流程**：与场景 1 完全相同——没有任何一步出门店。

**实现**：模型、产品库、权重、规范库全在盒内；SI app 若也部署在盒上（业务 app 上盒模式），连终端 UI 都不依赖外网。装机验收就是**拔网线跑完整评估**。期间总部改的权重下不来（用已装版本），网络恢复后 updater 自动追平（场景 6）。

## 场景 3 · 检测报告解读与复检对比（二期）

**看到什么**：顾客带来检测机构出的肠菌报告（纸质/PDF）。店员用工作台拍照上传，几秒后屏幕出现结构化指标卡：多样性指数、优势菌属、失衡菌属，超范围的标红并附通俗说明和科普；再点「对比」，和她三个月前那份报告的趋势变化并排展示（"双歧杆菌丰度回升……"），店员照着讲。

**流程**：

1. 上传报告图片 → ① chat 多模态（vision）做字段/指标抽取，`json_schema` 出结构化指标（OCR spike 达标走盒内；不达标 SI 接 OCR 服务兜底）；
2. SI app 用**阈值表（资产）**判定每项指标正常/偏高/偏低——规则判，不是模型判；
3. 把「指标 + 判定结果」送 ① chat：生成通俗摘要、异常说明、科普文案（规范库约束措辞）；
4. 对比：SI 从档案取该客户历史报告的结构化指标，连同本次一并送 ① chat 生成趋势解读；
5. 结构化报告 + 解读文本存档；指标自动填入该客户下次评估的 `report?` 块（推荐理由的"菌属匹配"从此有真数据）。

**实现**：消费端有两个（设备端「肠菌报告」模块 + 端二工作台），**同一能力线，盒子只建一次**。依赖：vision spike（P0，即刻）、T1.5 结构化保障（行业增量，落地前不作为 `contract-v1.0` 承诺）。

## 场景 4 · 店员下班前批量处理（二期）

**看到什么**：晚上 9 点闭店，店员把白天积累的 18 份报告一次性上传，点「批量解读」，锁门走人。早上开店，18 份结构化解读全部就绪，OCR 识别的姓名已自动匹配到客户档案。

**流程**：

1. SI app 逐份调用解读链路（场景 3 的 1~3 步）；T2 行业增量落地后，请求头带 **`X-Pinea-Priority: batch`**；
2. 设备闸门把 batch 放入低优先队列串行消化；若队列满返 `429 queue_full` + `Retry-After`，SI 客户端按头重试（quickstart 给样例代码）；
3. 期间若有顾客在终端做评估（interactive，缺省档），插队优先，批量任务让路；
4. 完成结果逐份回写档案，OCR 姓名字段做客户匹配（低置信度的标"待人工确认"）。

**实现**：QoS 优先级头（build-plan 并入 T2，P1，尚未进入 `contract-v1.0` Stable 矩阵）。落地后把「交互类>批量类」从内部原则变成 SI 可声明的接口语义——白天店员当面交互永远不被夜间/批量任务挤占。

## 场景 5 · 消费者自助百问百答（二期）

**看到什么**：顾客在等店员时，自己在终端上问："益生菌和酸奶有什么区别？要吃多久？"屏幕给出通俗、合规的回答，末尾附知识来源。她接着问"我在吃华法林能不能吃这个"——屏幕回答："涉及用药请咨询店内营养师"，并呼叫店员。

**流程**：

1. 问题 → SI app 先过**拒答边界**（客户圈定的清单：用药相互作用、疾病诊断类 → 直接转店员，不进模型）；
2. 通过的：① `/v1/embeddings` 向量化 → SI 侧向量库检索知识库 top-k；
3. 「问题 + 检索片段」送 ① chat，规范库约束生成，要求引用来源；
4. 输出过禁用词扫描（违例则不展示、转店员）→ 渲染答案 + 来源。

**实现**：与端二「售后问答」**同一条 RAG 能力线，两档护栏**——店员版有人把关，护栏低一档；消费者版拒答边界 + 禁用词扫描双闸。知识库与边界清单都是客户资产。

## 场景 6 · 总部上新调权重，三天到全国门店（三期）

**看到什么**：总部本月主推新品 GH-021，运营在总部后台把它加进产品库并调高权重。三天内，全国门店的终端在匹配症状的顾客评估里开始优先出现 GH-021；某偏远门店当周断网，恢复联网当天自动追平。总部月底对账：每条推荐记录都带资产版本号，新权重生效前后的推荐分布一目了然。

**流程**：

1. 总部后台（SI）编辑产品库/权重 → CI 打出 `gut-recommend v1.3` 资产包（含 hash）；
2. 各门店 updater（T-D #6 工具）定时拉取 → 校验 hash → 调盒子 ③ `POST /v1/skills/install` 重装（幂等，失败回滚旧版）；
3. 重装即生效：下一次评估用新权重；断网门店继续用 v1.2，恢复后追平；
4. 每条推荐存档带 `asset_version: v1.3` → 总部对账与效果归因。

**实现**：③ skill 版本化（registry 带 `version`/`hash`，build-plan 并入 T3）+ updater。**总部→多门店资产分发是连锁行业通用拓扑**，本场景跑通即成平台资产。

## 场景 7 · 通话录音自动归档（三期）

**看到什么**：运营系统昨晚 AI 外呼了 40 位意向客户。今早门店店长打开工作台，每位客户档案里已就绪：通话转写全文 + 提取摘要（核心需求、痛点、意向产品、到店意愿），按意向度排好序，店长照单跟进。

**流程**：

1. 外呼在云端发生（盒外，SI 运营系统），录音落云端；
2. 门店相关客户的录音回传该店盒子（或就近云端转写，三期定）→ ① `/v1/audio/transcriptions`（batch 档，夜间消化；>30min 切分）；
3. 转写文本 → ① chat + `json_schema` 提取 {核心需求, 痛点, 消费诉求, 意向产品, 到店意愿}；
4. 结构化结果回写客户档案（SI），按到店意愿排序生成跟进清单。

**实现**：T1 STT + 结构化提取 + batch QoS 的组合应用。外呼本身、电话线路、实时对话均盒外——盒子的角色是**通话后**的转写与理解。
