# 肠菌方案 · 一期设计：评估推荐线 + 设备端落地

> 前置：总方案见 [`integration-plan.md`](integration-plan.md)（责任边界、能力线映射、A/B 方案定义）。契约基线 `contract-v1.0`。
> 读者：我方 + SI。2026-06-11。

## 0. 一期为什么这么切，关键路径在哪

- **价值最大**：评估推荐线是设备的 AI 核心（客户原文"端侧大模型为本端核心"），P0 最密集——先交它，价值感知最强。
- **平台依赖最小（关键事实）**：全链路只依赖 ① `/v1/chat/completions` + structured outputs，**T0 已实现、conformance 已绿，零新端点依赖**。T1–T3 照 build-plan 并行推进，与一期互不阻塞，二三期再消费。
- **本详设的合同假设**：AI 能力线我方交付，业务 app（CRM/会员/支付/内容，约占设备功能 2/3）由 SI/客户开发——**app 开发归属是头号商务问题（requirements §5#1），M0 必须拍板**；客户要整包则业务 app 另立项报价，不挤本里程碑。
- **因此一期关键路径不是写代码，是两件事**：① 客户数据资产到位（§4，没有它样例集建不起来，所有效果承诺都是空话）；② 交付工程（T-D 清单在真实门店第一次落地）。排期围绕这两件事设闸门。

## 1. 一期范围（按能力线切）

| 线 | 进一期 | 留二/三期 |
| --- | --- | --- |
| **AI 能力线（我方）** | **评估推荐能力线**全链路（A 方案）：问卷+体检数据输入 → 评估/推荐/科普结构化输出；schema 加 `checkup?` 可选块（体检数据，字段待 §5#11 澄清） | 报告解读+**纵向对比**、百问百答（消费者版护栏）、报告 OCR——二期一并建，设备端与店员端共用 |
| **业务 app 线（SI/客户）** | 设备端 UI：问卷流程、推荐展示、店员调整/存档；CRM 建档（推荐结果挂档案） | 会员/支付/核销、内容运营（科普文章、产品/菌株介绍页）——纯业务，与 AI 无耦合，节奏自定 |
| **交付工程（我方）** | T-D 交付件首次实战：mock、quickstart、出厂脚本（含本方案样例）、诊断包、装机清单；**宿主环境口径**（若 app 上盒：进程/端口/资源配额约定） | 多门店复制（三期） |

一期走 **A 方案**（`integration-plan.md` §3.1）：SI 组装资产进 prompt，盒子出结构化结果。资产从第一天就版本化（§4），二期打包成 skill 时原样平移。

## 2. 评估推荐流水线（A 方案细化）

```
 [SI 应用]                      [Pinea 盒 :18800]                [SI 应用]
 问卷 UI → 问卷 JSON              ① chat + json_schema            店员确认/调整 UI
 ├ 红旗规则先判（规则表）   ──────→  system: 任务规约+规范库要点      ├ 绑定客户档案存档
 └ 规则预筛候选产品 ≤15            user: 问卷+候选+权重规则    ──→   └ 档案含 asset_version
                                 流式输出（先结论后推荐）
```

三步责任与设计理由：

1. **SI 侧前置规则（资产驱动，执行在 SI）**：
   - **红旗规则先于一切**：问卷命中报警项（如便血、不明原因消瘦等，红旗清单在规则表资产里）→ 直接输出「建议就医」，**不进推荐流程**。健康场景的安全底线必须是确定性规则，不过模型。
   - **候选预筛**：按症状/调理需求规则筛出候选产品 ≤15 个再送模型。理由：候选越收敛模型排序越稳、prompt 越短推理越快；预筛规则本身是资产（§4），二期平移进 skill 工序。
2. **盒内单次调用出全量结构化结果**：① chat，`response_format: json_schema`（schema 见 §3），SSE 流式让终端先展示评估结论、再逐条出推荐——到店交互体验预算见 §5。
3. **SI 侧调整与存档**：店员替换/补充推荐留痕；存档块（模型输出自带）+ 问卷快照 + 调整记录绑定客户档案。**每条存档带 `asset_version`**——产品库/权重哪个版本给出的推荐，可追溯可对账。

## 3. 接口形状（schema v1 提案，阶段 M0 与 SI 定稿后冻结）

演进规则与契约一致：**v1 冻结后只加可选字段，不改不删**。

### 3.1 评估请求（问卷 JSON，user 消息内容）

| 块 | 字段要点 |
| --- | --- |
| `profile` | 年龄段、性别、身高体重（基础信息） |
| `medical_history` | 疾病史[]、家族病史[]、用药与补充剂史[] |
| `gut_symptoms` | 症状清单（便秘/腹泻/腹胀…，含频率/持续时长） |
| `goals` | 调理需求[]（客户原文：便秘/腹泻、疾病风险、营养缺乏、调理需求） |
| `checkup?` | 可选：体检数据（设备版新增维度，字段范围/来源待 requirements §5#6 澄清；schema 留可选块，澄清后只增不改） |
| `report?` | 可选：已有肠菌报告关键指标（二期端二打通后自动带入，一期可人工录入或缺省） |
| `candidates` | SI 预筛后的候选产品[]（含 SKU、关联症状/菌属、禁忌）+ 权重规则要点 |

### 3.2 评估结果（json_schema 强约束输出）

| 块 | 字段要点 | 对应客户需求 |
| --- | --- | --- |
| `conclusion` | 评估综述 + 风险点[] | 肠道健康评估结论 |
| `red_flag` | `{triggered, advice}`（SI 已先判，模型复述就医建议口径） | 安全底线 |
| `recommendations[]` | `{sku, rank, reasons:{symptom_match[], flora_match[]}, cautions}` | 推荐结果 + 推荐理由（症状匹配+菌属匹配） |
| `education[]` | `{topic, highlight, text}` | 异常指标科普 + 知识点高亮 |
| `archive` | `{asset_version, model, schema_version}` | 可存档结构化数据 |

**Schema 合规率是可以 100% 承诺的**（structured outputs 是契约行为）；推荐质量只承诺样例集实测数字（§6）。

## 4. 资产五件套（一期就版本化，二期原样打包成 skill）

放本目录 `assets/`（即 README 规范里 `skills/` 的前身），单调版本号，进每条输出的 `archive` 块：

| 资产 | 内容 | 来源 |
| --- | --- | --- |
| 产品库 | SKU、症状/菌属关联、禁忌 | 客户提供（M0 给模板） |
| 权重规则 | 基础权重 + 排序策略 | 客户提供现行规则 |
| 规则表 | **红旗清单** + 候选预筛规则 + 指标阈值 | 客户口径 + 双方评审 |
| 回答规范库 | 话术、措辞边界、**禁用词表**（诊断/治疗类词） | 客户提供 + 合规口径（integration-plan §7） |
| 金标准样例集 | ≥20 份问卷 + 人工认可的推荐结果，**其中 ≥3 份红旗例** | 客户/双方共建 |

> 这五件套是一期最大的外部依赖。M0 第一周就把**模板表格**发给客户，按周催收——资产不到位，M1 之后全部停摆。逐件的字段级说明与覆盖要求见 [`customer-assets.md`](customer-assets.md)（可直接发客户）。

## 5. 模型选型与体验预算

- **候选模型**：qwen3 / gpt-oss:20b（盒子算力档位待澄清 requirements §5#3，是前置阻塞项）。**选型不拍脑袋：M1 用金标准样例集对候选模型打分，谁的实测分高用谁。**
- **体验预算**：到店同步交互，目标**单次评估 P95 ≤ 30s**；SSE 流式先出 `conclusion` 后出 `recommendations`，终端可边收边展示。评估模型 `keep_alive` 常驻（交互类 QoS，`model-gateway.md` §6.4）。
- **断网即验收项**：装机验收含**拔网线跑完整评估**——「离线推理引擎 P0」用这一条实测交差，不用嘴说。

## 6. 一期就绪（五条全绿才算交付，对齐 build-plan「首单就绪」）

1. **schema v1 双方签认**，mock 模式返回样例结果，SI 凭 quickstart 30 分钟跑通评估调用；
2. **金标准样例集 ≥20 落库**（含 ≥3 红旗例），评分口径双方签认（推荐与人工认可结果的重合标准）；
3. **真模型实测达线**：schema 合规 100% + 推荐质量 ≥ 双方约定线（如 18/20，承诺实测数字）+ **红旗例 3/3 全部触发就医建议且零产品推荐** + 禁用词扫描 0 违例（脚本自动扫输出）；
4. **断网全流程绿** + 出厂验收脚本（conformance + 端口自检 + 本方案样例集）一键绿；
5. 诊断包一条命令出包，装机/加固清单在首店落地。

## 7. 里程碑（按周相对排，客户资产到位是 M1 闸门）

| 里程碑 | 内容 | 出口标准 |
| --- | --- | --- |
| **M0（第 1 周）** | schema v1 提案评审定稿；mock + quickstart 交 SI；**资产模板表格交客户**；**商务前置三件拍板（requirements §5#1–3：app 归属、「诊疗方案」改口径、硬件档位）**；体检数据范围澄清 | SI 对 mock 跑通；客户开始填资产 |
| **M1（资产到位 + 2 周）** | 资产五件套入库（v1）；prompt 组装规约定稿；候选模型样例集打分→**选型拍板**；首测出数 | 实测有数字（哪怕不达线） |
| **M2（+2 周）** | 调优循环：规范库/权重/预筛迭代 → 复测；验收脚本固化；禁用词扫描接入 | §6 第 3 条达线 |
| **M3（+1~2 周）** | 首店装机试点：T-D 清单全套落地，店员真实使用 ≥1 周收反馈 | §6 五条全绿，一期验收 |

SI 并行线：M0 起即可用 mock 开发终端 UI 与存档/调整功能，不等模型侧。

## 8. 风险与对策

| # | 风险 | 对策 |
| --- | --- | --- |
| 1 | **客户资产滞后（最大风险，在关键路径上）** | M0 就给模板 + 示例行；按周对账；缺哪件就明说哪个里程碑顺延 |
| 2 | 中文健康域推荐质量不达线 | 三层兜底：换候选模型 → 收紧预筛（候选越少越稳）→ 收敛推荐数到 top-3；规范库迭代是常规调优手段 |
| 3 | 措辞合规翻车 | 双保险：规范库约束生成 + 验收/上线后 SI 展示层禁用词扫描（确定性，不赌模型） |
| 4 | 硬件档位未定阻塞选型 | M0 必须拍板（requirements §5#3）；先按两档各备一个候选模型 |

## 9. 二期衔接（一期里埋好的钩子）

- 资产五件套版本化 → 二期打包成 `gut-recommend` skill（③），调用从 A 切 B 时 **SI 只改请求体**（②+`skill` 字段），schema 不变；
- `report?` 可选块 → 二期端二报告解读打通后自动填入，推荐理由的「菌属匹配」从可选变常态；
- `archive.asset_version` → 三期总部下发（skill 包版本发布）的对账基础。
