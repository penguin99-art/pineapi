# 肠菌方案 · 客户资产清单（一期 · 发给客户用）

> 配套：一期设计见 [`phase1-design.md`](phase1-design.md) §4。本清单是 M0 第一周要交到客户手上的催收依据。
> 原则：**这五件资产是 AI 推荐的"原料"，缺哪件，对应的里程碑就顺延哪件**。我方提供 Excel 模板与示例行，客户按模板填。

## 总表

| # | 资产 | 形式 | 谁来出 | 没有它会怎样 |
| --- | --- | --- | --- | --- |
| 1 | 产品库 | Excel（我方给模板） | 客户产品/运营 | 无候选可推，链路无法启动 |
| 2 | 权重规则 | 文字说明或表格 | 客户运营负责人 | 推荐排序没有依据，只能模型自由发挥 |
| 3 | 规则表（红旗清单 + 预筛规则） | Excel + 文字口径 | **客户专业人员（营养师/医学顾问）签字确认** | 安全底线缺失，不能上线 |
| 4 | 回答规范库 | 文字文档 + 禁用词表 | 客户品牌/合规 | 措辞合规无法兜底 |
| 5 | 金标准样例集 ≥20 份 | 问卷+人工认可结果（我方给模板） | 客户资深店员/营养师 | **没有验收基准，一切效果承诺都是空话** |

## 1. 产品库（全量在售 SKU，逐行填写）

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| `sku_id` / 产品名 | 唯一编号 + 名称 | GH-001 / 复合益生菌粉 |
| 类别 | 益生菌 / 益生元 / 膳食纤维 / 复合营养 / 其他 | 益生菌 |
| 规格与服用方式 | 含量、频次、疗程 | 2g×30 袋，每日 1 袋，4 周 |
| **适用症状[]** | 从受控词表选（便秘/腹泻/腹胀/排气/口臭/食欲差…，词表双方 M0 定） | 便秘、腹胀 |
| **关联菌属[]** | 该产品针对哪些菌属失衡（推荐理由的"菌属匹配"来源） | 双歧杆菌偏低、拟杆菌偏高 |
| 适用调理需求[] | 对应问卷的调理需求项 | 改善排便、营养补充 |
| **禁忌** | 人群（孕妇/儿童/免疫低下）、药物相互作用、过敏原 | 含乳制品，乳糖不耐慎用 |
| 推荐话术要点（可选） | 一两句卖点，供生成理由时引用 | 临床菌株 XX，餐后服用 |

> 适用症状、关联菌属、禁忌三列是推荐质量的命脉，**宁可少而准，不要多而糊**。

## 2. 权重规则（现行运营规则的书面化）

要回答四个问题：

1. **基础权重**：每个 SKU（或每类别）的优先级分值/分档（主推/常规/清库）？
2. **排序逻辑**：症状匹配度与运营权重怎么平衡？有无季节/库存/利润因素？
3. **组合规则**：一次最多推荐几个？是否"主推 1 + 搭配 1~2"？哪些产品互斥/不可同推？
4. **例外**：有没有"某症状必推某品"或"某人群禁推某品"的硬规则？（硬规则会进规则表确定性执行，不交给模型）

> 口头规则也行，访谈一次由我方整理成表、客户确认。三期总部下发功能改的就是这份。

## 3. 规则表（安全底线，需专业人员签字）

### 3.1 红旗清单（最高优先级）

| 列 | 说明 |
| --- | --- |
| 触发条件 | 问卷答案的组合，如：便血 / 黑便 / 6 个月内不明原因体重下降 >5kg / 持续发热 + 腹痛 / 肿瘤家族史 + 报警症状 |
| 处置 | 一律：终止推荐流程，输出就医建议 |
| 就医建议话术 | 客户认可的标准措辞（不引发恐慌、不做诊断） |

**这份清单由客户的医学/营养专业人员给出并签字**——它是确定性规则直接执行，不过模型，也是我们能对"安全"做硬承诺的唯一方式。

### 3.2 候选预筛规则

症状/调理需求 → 产品类别的映射表（如：便秘 → 益生菌+膳食纤维类；腹泻 → 特定菌株类），用于把候选收敛到 ≤15 个再送模型。

### 3.3 肠菌指标阈值表（一期可后置，建议一并给）

各指标（多样性指数、主要菌属丰度）的正常范围、偏高/偏低判定、对应异常说明口径。一期问卷流程可选用，**二期报告解读是必需品**，早给早省事。

## 4. 回答规范库（品牌与合规口径）

1. **风格**：称呼、语气、推荐理由的详略偏好（给 2~3 段客户满意的范文最有效）；
2. **措辞边界**：能说（调理/改善/辅助/参考）vs 不能说（治疗/治愈/诊断/疗效保证）；
3. **禁用词表**：逐词列出，我方做成自动扫描（验收项：0 违例）；
4. **固定文案**：免责声明、就医建议结尾语等必须原样出现的句子；
5. **标准答法示例 5~10 条**：常见场景（如"益生菌要吃多久"）的认可答法。

## 5. 金标准样例集（≥20 份，验收的标尺）

每份 = **完整问卷答案 + 资深店员/营养师人工认可的推荐结果**（推荐哪些产品 + 为什么）。

覆盖要求：

| 类型 | 份数 | 说明 |
| --- | --- | --- |
| 红旗例 | ≥3 | 验收要求 3/3 触发就医建议且**零产品推荐** |
| 常见单一症状（便秘/腹泻/腹胀各） | 各 ≥3 | 主流场景 |
| 混合症状 / 有疾病史或用药史 | ≥4 | 考察禁忌处理 |
| 无明显症状、仅调理需求 | ≥2 | 考察不过度推荐 |
| 不同年龄段分布 | 穿插 | 老人/儿童相关禁忌 |

同时双方签认**评分口径**：模型推荐与人工认可结果怎么算"对"（如：top-3 与人工重合 ≥2 算通过）。这套样例集 = 模型选型打分依据 = 一期验收基准 = 二期 skill 验收集，一次投入三次复用。

引 [`requirements.md`](requirements.md) §5 商务前置三件：终端业务 app 开发归属（#1）、「出具诊疗方案」改口径（#2）、硬件档位/预算（#3，决定模型选型）；另尽早明确三端部署形态（#4）与合规审查责任（#13）。

## 数据安全承诺（写给客户看）

所有资产仅进入门店本地盒子（离线推理），不上云、不出门店；版本化管理（`asset_version`），每条推荐可追溯到资产版本；我方对资产内容承担保密义务（按合同）。
