Pinea / Piny 架构与接口总览

本文基于当前已落盘的架构、Gateway、ToB 接口与 ToC Piny 讨论整理而成。 目标是把系统边界、ToB 对外接口、ToC Soul 支撑逻辑和实现约束放在一张可阅读的说明里。

快照（2026-06-10）：本 HTML 是汇总视图，权威以 docs/architecture.md 等 Markdown 契约为准；如二者不一致以 Markdown 为准。

1. 一句话架构

Pinea Core 是可发 B 端的无头能力底座，包含模型网关、agent runtime、白盒记忆、工具、MCP、skills 与 workspace。 Piny toC 是 C 端专属灵魂层，在 Core 旁边增加感知、Soul、表达和 PineaState 总线，负责身体、关系和主动性。

蓝色 / Core

Pinea Core

可无头部署，可发 B 端。只提供模型、agent、记忆、工具、skills、数据与安全边界，不包含认主、陪伴、灯带等 C 端灵魂逻辑。

橙色 / Soul

Piny 灵魂层

C 端专属。负责感知事件、主人关系、主动时机、6 态状态机和多端表达，只通过契约调用 Core，不 import Core 内部模块。

红线：不改 PilotDeck 核心；Core 不 import 感知/表达/Soul；Studio 只依赖 gateway 契约；表达端只订阅状态，不自行判态。

记忆也分层：Core memory（白盒记忆）存对话/知识/任务/workspace，随 Core 发 B 端；Soul memory存主人特征、关系、节律、打扰偏好，C 端专属、绝不进 Core。Core 永不读 Soul memory。

3. ToB 架构：本机设备前门

当前 ToB 不是云端 API 服务，而是卖一台装好 Pinea Core 的松果派设备。 SI App / 容器优先部署在盒子本机，经 127.0.0.1:18800 调 Pinea Model Gateway。 如需绑定 LAN/WAN，必须开启 Bearer Token。

SI 集成
SI App / 容器 → localhost → Pinea Model Gateway。业务登录、最终用户权限、行业权限由 SI App 自己负责。
设备门锁
Authorization: Bearer <token>。本机封闭部署可关闭；绑定 LAN/WAN 时 PINEA_API_KEY 必填，否则 Gateway 拒绝启动。
非目标
MVP 不做多租户、商业计费、配额、购买范围控制。usage 只用于本机观测、排障和资源仲裁。

三类 ToB 接口

类卖什么入口状态实现落点
① 能力面 算力 + 多模态 /v1/chat、/v1/audio、/v1/images、/v1/video、/v1/embeddings 无状态 Gateway → 模型后端 adapter
② Agent 面 开箱即用 agent /v1/agent/* + X-Session-Id 有状态 Gateway → Runtime api_server
③ Skills 可复用行业能力 /v1/skills/* 管理面 Gateway 管理接口 → runtime skill 目录 → agent 自动触发

4. ToC Piny：Soul 如何支撑

ToC Piny 不重做 agent runtime。它复用 Core 的 agent、memory、tools 和 skills，在旁边增加一套 C 端专属的 Soul 系统。 Soul 的职责是判断人、关系、时机、打扰边界和表达状态。

输入端桌伴 / Studio / Phone / 外设 / 麦克风 / 摄像头
→
感知VAD / motion / ASR / 视觉理解 / 认主
→
PineaState事件总线，承载 presence、voice、intent
Soul 反射层规则/fast model 路由：ignore / listen / reply / escalate
→
Core Agent（仅 escalate）复杂任务、工具、skills、白盒记忆
→
表达只订阅 state：灯带 / TTS / 桌伴 / Phone / Studio

实时响应路由（快/慢分流）

ToC 不把每个事件都丢进 agent。Soul 先用规则或 fast model 做低延迟判断；只有需要长上下文、记忆、工具或多步时才升级（escalate）进 Agent Runtime。 fast model 不是第二套 agent，不调工具、不写 Core memory。

判断延迟目标动作进 Agent Runtime？
ignore近实时不响应，只更新 presence/节律否
listen近实时切 listening，等用户继续说否
reply低延迟fast model / 模板生成短回复，表达层播报否
escalate可慢Soul 调 submitTurn 进入 agent loop是

Soul 的唯一状态机

状态含义典型表达
idle空闲，等待事件低亮度或待机
listening正在听用户说话收音灯效、录音提示
thinkingCore 正在推理或规划思考动效
speaking正在播报或输出TTS、字幕、桌伴口型
working正在执行工具或长任务任务进度、工具执行提示
error失败或需要用户介入错误提示、降级表达
关键约束：只有 Soul 状态机可以写最终 state。表达层、Studio、桌伴、Phone、外设都只能订阅并渲染，不能自行判断当前应当是什么状态。

5. 进程与运行边界

进程职责边界
P1 Pinea Model Gateway FastAPI 设备前门，OpenAI 兼容端点、Agent 转发、Skills 管理、发现、健康、日志。 Python；只路由和归一，不做推理。
P2 Agent Runtime agent loop、白盒记忆、tools/MCP、skills、workspace、智能路由。 当前实现 PilotDeck；只走扩展点，不改核心。
P3 感知服务 采集和归一 presence、voice、intent 事件。 不直接驱动表达，不直接改 Core memory。
P4 Soul 服务 消费感知和 core.lifecycle，判断主动时机，必要时 submitTurn，写唯一 state。 C 端专属，不进入 Core。
P5 表达服务 订阅 state，渲染灯带、TTS、桌伴、Studio、Phone 和外设反馈。 无状态决策权，只做渲染。

6. 对外接口

6.1 共用约定

项目约定
Base URLhttp://<host>:18800，默认绑定 127.0.0.1。
鉴权/门锁Authorization: Bearer <token>。本机封闭可关闭，外放必须开启。
发现GET /v1/capabilities 返回模态、agent、skills 可用性。
模型GET /v1/models 聚合后端模型列表。
健康GET /healthz 聚合后端健康状态。
错误形状OpenAI 风格：{ "error": { "message", "type", "code" } }。

6.2 ① 能力面

端点能力默认后端说明
POST /v1/chat/completions文本 / 工具 / 视觉ollamaOpenAI Chat Completions 兼容；tool calling 已验证。
POST /v1/embeddings向量ollama透传 nomic-embed 等 embedding 模型。
POST /v1/audio/transcriptionsSTTspeaches / faster-whisperMVP 首做；multipart/form-data。
POST /v1/audio/speechTTSPiper / Kokoro表达层也可复用。
POST /v1/images/generations生图ComfyUIadapter 把工作流封装成 OpenAI 风格响应。
POST /v1/video/generations生视频ComfyUI自定义异步任务：submit → poll / webhook。

6.3 ② Agent 面

端点说明
POST /v1/agent/chat/completions OpenAI Chat 兼容 + X-Session-Id。Gateway 内部翻译到 runtime session，一次请求由 agent 自己完成多步工具调用。
GET /v1/agent/sessions/{id}/messages 只读会话历史，供 UI 回放。
GET /v1/agent/sessions 列出会话。

同一 session 同时只允许一个进行中回合，并发返回 429 rate_limit_error。 默认 workspace 应配置到设备 ToB 工作区，例如 /srv/pinea/workspaces/default，不得暴露系统根目录。

6.4 ③ Skills 管理

操作端点说明
列表GET /v1/skills返回已安装 skill、启停状态、描述。
安装POST /v1/skills/install上传 zip/tar 或提交 local_path；校验 SKILL.md 后落盘。
读取GET /v1/skills/{name}读取 SKILL.md 与元信息。
启用POST /v1/skills/{name}/enable允许 agent 自动触发。
禁用POST /v1/skills/{name}/disable保留文件但不触发。
删除DELETE /v1/skills/{name}移除 skill，需保持 registry 和磁盘一致。
扫描POST /v1/skills/scan调试/装配时重新扫描底层目录。

8. 安全、观测与资源仲裁

安全门锁

默认 PINEA_HOST=127.0.0.1。绑定到 0.0.0.0、LAN IP 或公网时，PINEA_API_KEY 必填。

观测日志

每请求记录 caller、capability、backend、usage、latency。MVP 不承载商业计费语义。

资源仲裁

chat/embedding 常驻；STT/TTS 优先 CPU/小显存；生图/视频/vllm-omni 等重后端默认排队或按需起停。

9. 实现落地检查表

  • Gateway 默认只监听 127.0.0.1。
  • Gateway 外放时必须配置 Bearer Token。
  • 三类 ToB 共用 /v1/capabilities、错误形状和 usage 日志。
  • 能力面 adapter 只做 schema 归一，不做推理。
  • Agent 面只转发到 runtime api_server，不绕过 runtime 权限。
  • Skill 通过接口安装、启停、删除，磁盘目录是内部实现细节。
  • Skill 只依赖 Gateway 能力端点和 runtime skill/tool 契约，绝不依赖 Soul。
  • PineaState 承载 presence、voice、intent、core.lifecycle、state。
  • 只有 Soul 状态机可以写最终 state。
  • 表达层只订阅 state，不自行判断状态。
  • Core 不 import 感知、表达或 Soul。
  • 不修改 vendor/pilotdeck 核心代码。
